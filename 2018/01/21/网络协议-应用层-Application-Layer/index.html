网络协议-应用层(Application Layer)

发表于 2018-01-21 | 分类于 Network | | 阅读次数:

应用程序收到”传输层”的数据，接下来就要进行解读。由于互联网是开放架构，数据来源五花八门，必须事先规定好格式，否则根本无法解读。
“应用层”的作用，就是规定应用程序的数据格式。
举例来说，TCP协议可以为各种各样的程序传递数据，比如Email、WWW、FTP等等。那么，必须有不同协议规定电子邮件、网页、FTP数据的格式，这些应用程序协议就构成了”应用层”。
这是最高的一层，直接面对用户。

一、DNS(Domain Name System, 域名系统)

因特网上作为域名和IP地址相互映射的一个分布式数据库，通过域名得到对应的IP地址的过程叫做域名解析。DNS协议运行在UDP协议之上，使用端口号53。

域名到IP地址的映射有两种方式：

  1. 静态映射，每台设备上都配置主机到IP地址的映射，各设备独立维护自己的映射表，而且只供本设备使用(如Windows系统hosts文件)；
  2. 动态映射，建立一套域名解析系统（DNS），只在专门的DNS服务器上配置主机到IP地址的映射，网络上需要使用主机名通信的设备，首先需要到DNS服务器查询主机所对应的IP地址。

在解析域名时，可以首先采用静态域名解析的方法，如果静态域名解析不成功，再采用动态域名解析的方法。可以将一些常用的域名放入静态域名解析表中，这样可以大大提高域名解析效率。

DNS冗余：为保证服务的高可用性，DNS要求使用多台名称服务器冗余支持每个区域。同一个区域中，有一台 主DNS服务器 和多个 辅DNS服务器，辅DNS服务器用作备份，以防主服务器无法访问或宕机。辅DNS服务器定期与主DNS服务器通讯，确保它的区域信息保持最新，如果不是最新信息，辅DNS服务器就会从主服务器获取最新区域数据文件的副本。

二、Email

因为用户电脑性能的限制，无法运行收发邮件的程序，并且无法不间断地运行并连接到互联网上，故在邮件是通过邮件服务器来真正传输的。

用户代理(user agent) 就是我们常见的邮件客户端，邮件传输代理(message transfer agent) 非正式地称为邮件服务器。
邮件服务器之间的 SMTP(Simple Mail Transfer Protocal, 简单邮件传输协议) 协议使用TCP连接，可以保证邮件准确无误地在邮件服务器之间传输。

2.1 邮件格式

RFC 5322：在Email初期，除了必须的主题和收件人，内容只能发送 ASCII 文本消息。
MIME(Multipurpose Internet Mail Extensions, 多用途Internet邮件扩展)：随着Internet在全球范围得到广泛使用，满足不了使用其他语言撰写的邮件(如中文和日文)以及完全不包含文本的邮件(如图片、视频或程序)。这也是我们现在所使用的邮件格式。

在Content-Type字段中描述文件类型。

2.2 交互协议

邮件服务器之间使用 SMTP 协议来传输邮件，在邮箱客户端和邮件服务器之间也需要商定一种协议。
POP3(Post Office Protocal, 邮局协议版本3)：这是最初的交付协议，功能简单，不太安全。它通常把投建下账到用户计算机上，而不是留在服务器上，虽然减少了服务器的压力，但不能同步邮件，并且一旦删除邮件或计算机发生故障，所有的电子有限将可能永久丢失。
IMAP(Internet Message Access Protocal, Internet邮件访问协议)：它是POP3的升级版，功能丰富，并有安全机制。为了使用IMAP，邮件服务器必须运行IMAP服务器，它负责监听端口143，同时用户代理必须运行IMAP客户端，邮件默认是在IMAP服务器存放着，用户代理可以选择下载邮件或直接查看，这样就实现了同步的功能。
Webmail：让每个用户都下载一个邮件客户端还是有些麻烦的，目前流行的提供电子邮件服务的是Webmail，利用Web为用户提供邮件服务。选择广泛使用的Webmail系统包括 Google Gmail、MS Outlook和Yahoo Mail等。

三、HTTP(HyperText Transfer Protocal, 超文本传输协议)

如果想在网上冲浪，我们就必须打开浏览器输入对应网站的 URL(Uniform Resource Locator, 统一资源定位符)。用户请求和看到的页面使用的是 HTML(HyperText Markup Language, 超文本标记语言)，使用的浏览器就是一个HTML解释器，称为Web客户端，它来代理用户通过HTTP协议和Web服务器进行交互，浏览器会与服务器的端口80建立一个TCP连接。
现在我们使用的浏览器能解释常见的文件类型，为了能解释快速增长的陌生文件类型，它会读取页面里的MIME类型，安装 插件(plug-in) 或 辅助应用程序(helper application)。为了使浏览器调用对应的插件，所有的插件必须实现该浏览器的一组过程程序；而辅助应用程序是一些独立于浏览器而存在的大型程序。
HTTP请求时一般常见的两种方法是GET和POST。GET方法通过URL传输明文参数，常用于搜索信息时；POST方法通过报文传输密文数据，常用于表单提交。
每个请求都会得到一个包含3个数字状态码和附加信息的响应消息。这个响应状态码分为5大组。

3.1 Cookie

浏览大多数网站时，浏览器和Web服务器之间使用 Session会话 模式，这样完成一次交互，用户下次请求时，服务器就完全分辨不出这个用户是谁，每次都当做新用户来看待。这对于用户购物的电子商务网站或个性化定制门户网站是不能接受的。
解决方案是采用了称为 小甜饼Cookie 的技术，它是一小段文本信息(最多4KB)，伴随着用户请求页面在服务器和浏览器之间传递。这个名称来自于很早以前的程序员行话，即程序调用一个过程并得到一些信息，这些信息稍后再完成某些工作时可能需要用的到。
当用户请求一个Web页面时，服务器除了提供所请求的页面以外，还以Cookie的形式提供了一些附加信息。浏览器把服务器提供的Cookie通常存储在硬盘对应Cookie目录下一段时间，这样在整个浏览器调用期间一直坚持Coookie，除非用户禁用Cookie。因为Cookie只是字符串不是可执行程序，所以即使一个Cookie包含病毒也只能被当作数据处理，因而不存在病毒得以实际运行从而造成损害的正式途径。所以你会发现清除浏览器的缓存后(包括Cookie)，某些用户的个人设置恢复成了默认设置。
一个Cookie至多5个字段。

以图片中第2行举例：顾客浏览 jills-store.com，当单击一件物品时，服务器将该物品放入该用户的购物车(由服务器Session维护)，同时创建一个包含商品数量和产品代码的Cookie，并把它发送给客户。每当顾客点击一个新商品，该Cookie随着该新页面的请求被返回到服务器端，服务器会将它们追加到Cookie中。最后该顾客进行结账时，这个Cookie就包含了想购买商品的完整列表，它和当前的请求一起发送给服务器。
利用Cookie，服务器很容易地就能跟踪用户的活动。

3.2 动态Web页面

如今静态页面已经不能满足日益增长的需求，为了处理动态页面，Web服务端采用 CGI(Common Gateway Interface, 公共网关接口) 和 内嵌的脚本(PHP,JSP,ASP)。
CGI提供了允许Web服务器和脚本通信的接口，这些脚本接受输入请求并生成HTML页面作为响应。对应的脚本语言有Python、Ruby和Perl等。 通过CGI调用的程序常驻在cgi-bin的目录下，该目录在URL中可以看到，服务器将一个请求映射到这个目录下的脚本并运行，该脚本处理数据并返回一个给浏览器的网页。
第二种方式是在HTML页面中嵌入少量脚本，然后服务器执行这些脚本并生成最终发送给客户的页面。编写这些脚本的一种流行语言是 PHP(HyperText Preprocessor, 超文本预处理器)，为了使用PHP，服务器必须能够理解PHP，就好像浏览器必须理解CSS才可以解释具有样式表的Web页面一样，通常服务器用文件扩展名php来标识包含PHP的Web页面，而不是用html。

而最流行的客户端脚本语言是 JavaScript。PHP和JavaScript都是嵌入在HTML文件中的代码，但它们的处理方式完全不同，PHP脚本是由服务器解释的，JavaScript脚本是浏览器解释的。

另外还可以在客户端执行的是 Applet 和 ActiveX。Applet就是Java小程序，被具有JVM能力的浏览器解释执行；ActiveX是一种被编译成x86机器指令的程序，可以直接在硬件逻辑上执行，故这种控件执行速度很快。所有浏览器都可以解释Java程序和JavaScript，故JavaScript和Applet的跨平台移植性好；JavaScript容易编写，Applet执行更快一点，ActiveX控件运行速度最快。

AJAX(Asynchronous JAvascript and Xml, 异步JavaScript和XML)：为了使用户无感地进行浏览器和Web服务器的交互，几种关键技术的组合称为AJAX。许多Web应用如Google的Gmail、地图和文档都是以AJAX编写的。
AJAX不是一种语言，是一组需要一起协同工作的技术：

  1. 直观展示页面的HTML和CSS；
  2. 改变页面结构的DOM(Document Object Model, 文档对象模型)。DOM可以表示结构化成一棵反映HTML元素的树，有根与叶、父节点和子节点等。DOM模型的意义在于它可以改变部分页面内容，没有必要重写整个页面，只有包含了需要改变内容的树节点才对页面内容进行替换。
  3. 浏览器和服务器传输数据的XML(eXtensible Markup Language)。XML相对于HTML更注重数据的表示，目前更流行的表示数据的结构语言有JSON和YAML。
  4. 发送和检索XML数据的异步HTTP通信；
  5. 将上述功能组合在一起的JavaScript；

四、内容分发

4.1 CDN(Content Distribution Network, 内容分发网络)

在国内访问Google美国服务器肯定不如肉身翻墙在USA访问快。Google为了很好的实现全球化，解决因物理距离导致的访问延迟，就需要在亚洲也设置服务器，如果你使用VPN上Google的话，请求默认会请求到香港或日本的服务器。

一般在下载大型文件的场景中，使用 镜像(mirror)，不同网络区域中CDN节点成为镜像点。这种设计让用户手动选择一个附近的镜像点来下载内容。
而在浏览网页时，大多数全球网站使用 DNS镜像(DNS redirection)。比如所有的用户在浏览器输入相同的www.cdn.com的URL时，DNS会根据用户位置把该域名解析为里用户最近的IP地址，这样不同地区的用户输入了同样的URL却请求的是不同的Web服务器。

4.2 P2P(Peer-to-Peer, 对等网络)

当然并不是所有的个人或组织都有资源搭建或租赁CDN服务器，这时就可以使用P2P。基本思想是将多台计算机连在一起并将资源集中起来形成一个内容分发系统，彼此相互服务提供内容，无须单独配置服务器或任何中央控制节点。
